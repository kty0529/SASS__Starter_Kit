# Repository Guidelines

이 스타터 키트는 반복 활용 가능한 SCSS 기본 자산을 정리한 저장소입니다. 아래 지침을 따르면 다수 프로젝트에서 동일한 헬퍼 세트를 안정적으로 재사용할 수 있습니다.

## 프로젝트 구조 및 모듈 구성
- `scss/base/`는 초기화, 타이포그래피, 컬러 부분 파일을 모아두며 모든 기능 스타일보다 먼저 로드해야 합니다.
- `scss/helpers/`에는 `_variables.scss`, `_mixins.scss`, `_index.scss`가 있으며 `@use "./helpers" as *`로 공용 토큰을 불러옵니다.
- 새 컴포넌트는 `scss/` 하위에 부분 파일로 추가하고 `core.scss`에 `@use`를 등록해 워처가 빌드하도록 합니다.
- 산출물 CSS는 `css/` 디렉터리에 생성되며 수동 수정 없이 Sass CLI가 덮어쓰도록 유지합니다.

## 빌드, 테스트, 개발 명령어
- `npm install`로 Stylelint와 Dart Sass CLI를 설치해 로컬 환경을 준비합니다.
- `npm run watch`는 `sass --watch scss:css`를 실행해 저장 시마다 SCSS를 CSS로 컴파일합니다.
- `npm run lint`는 Stylelint를 돌려 속성 정렬과 SCSS 규칙 위반을 검사하며, 직접 실행 시 `--fix` 옵션으로 자동 수정할 수 있습니다.

## 코딩 스타일 및 네이밍 규칙
- 들여쓰기는 공통으로 두 칸을 사용하고 블록은 멀티라인으로 작성해 가독성을 확보합니다.
- 믹스인과 함수 이름은 `px-to-vw`처럼 소문자-하이픈 패턴을 사용하고 파라미터 설명을 인라인 주석으로 남깁니다.
- 모든 부분 파일 앞에는 `_`를 붙이고 `core.scss`에서는 base → helpers → 기능 순으로 `@use`를 정렬합니다.

## 테스트 가이드라인
- 모든 변경은 최소 한 번 `npm run lint`를 통과해야 하며 경고를 로컬에서 해결합니다.
- 구조를 수정한 뒤에는 워처를 실행해 변경된 CSS가 `css/` 외부에 생성되지 않는지 확인합니다.
- 동작 변화가 있다면 PR 설명에 테스트한 테마나 브레이크포인트 등 수동 확인 내용을 기록합니다.

## 커밋 및 Pull Request 가이드라인
- Git 히스토리의 관례대로 `feat:`, `chore:`, `refactor:` 등 접두사를 사용하고 명령형 어조로 메시지를 작성합니다.
- 서로 다른 수정은 묶지 말고 믹스인, 변수 세트, 구조 변경 등 한 가지 주제로 커밋을 구성합니다.
- PR에는 영향 요약, 수정한 부분 파일 목록, 관련 이슈 링크, 시각적 변화가 있는 경우 스크린샷이나 코드 샘플을 포함합니다.

## AI 응답 및 처리 과정 안내
- AI 또는 자동화 도구를 활용할 때는 모든 응답과 처리 과정을 한국어로 작성해 팀원이 즉시 이해할 수 있도록 합니다.
- 생성된 코드나 문서를 반영하기 전에 한국어로 검토 로그를 남기고, 적용 여부를 PR 본문에 명시합니다.
- 외부 프롬프트를 사용할 경우 입력과 출력 요약을 한국어로 기록해 재현성을 확보합니다.
